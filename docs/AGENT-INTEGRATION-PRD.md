## 一、执行摘要

### 背景

SuperClaw101 是我们的 OpenClaw 知识中心网站（Astro + Starlight），已构建 77 页内容。当前网站以静态教程为主，缺少**互动体验**——访客看完教程后无法立即"感受" OpenClaw 的核心价值：AI Agent 的智能交互能力。

### 核心目标

```
将 OpenClaw Agent 试用体验嵌入 SuperClaw101
→ 让访客在 30 秒内体验 Agent 的真实能力
→ 将内容流量转化为产品用户
```

### 关键指标

| 指标 | 目标 | 衡量方式 |
|------|------|----------|
| 试用转化率 | >15% 访客点击试用 | 埋点统计 |
| 试用完成率 | >60% 完成至少 1 轮对话 | 会话日志 |
| 注册转化率 | >5% 试用后注册 | 注册漏斗 |
| 付费转化率 | >2% 注册后付费 | Stripe 数据 |

## 二、Agent 试用体验设计

### 2.1 交互模式选择

经过分析三种方案，推荐 **嵌入式 Chat Widget** 作为主方案：

| 方案 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **嵌入式 Chat Widget** | 体验流畅、不离开页面、转化率高 | 需前端开发 | ⭐⭐⭐⭐⭐ |
| 独立 Demo 页面 | 实现简单、可展示更多功能 | 跳出率高、打断阅读流 | ⭐⭐⭐ |
| 全站 AI 助手 | 全场景覆盖 | 开发成本高、可能干扰阅读 | ⭐⭐⭐ |

### 3.2 前端集成方案

#### Chat Widget 技术选型

```typescript
// 推荐方案: Web Component (框架无关，Astro 兼容)
<openclaw-chat
  agent-id="demo"
  theme="dark"
  position="bottom-right"
  welcome-message="你好！试试问我任何问题"
  max-tokens="5000"
  rate-limit="10"
></openclaw-chat>
```

**技术实现**:
```
Chat Widget (Web Component / React)
  ├── UI 层: 对话界面、消息气泡、输入框
  ├── 状态层: 会话管理、消息队列、token 计数
  ├── 通信层: WebSocket 客户端 / REST API 客户端
  └── 认证层: 匿名 token / JWT token
```

#### 连接方式

**方案 A: OpenAI-compatible HTTP API（推荐 Phase 2）**

```javascript
// 使用标准 OpenAI SDK 对接
const response = await fetch('https://api.superclaw101.com/v1/chat/completions', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({
    model: 'openclaw:demo',
    messages: [{ role: 'user', content: '帮我写一个快速排序' }],
    max_tokens: 5000
  })
});
```

**优势**: 标准 OpenAI API 格式，生态兼容性好，实现简单

**方案 B: WebSocket RPC（推荐 Phase 3）**

```javascript
// 完整 Gateway 协议，支持流式输出、工具调用、会话管理
const ws = new WebSocket('wss://api.superclaw101.com/ws');
ws.send(JSON.stringify({
  type: 'req',
  method: 'agent.run',
  params: {
    agentId: 'demo',
    prompt: '帮我写一个快速排序',
    sessionKey: 'trial-user-xxx'
  }
}));
```

**优势**: 流式输出、实时反馈、支持工具调用

### 3.3 后端架构

#### BFF (Backend for Frontend) 层

```
BFF Server (Node.js / Hono)
│
├── POST /api/trial/start     → 创建试用会话
├── POST /api/trial/chat      → 代理到 Gateway
├── GET  /api/trial/status    → 查询额度状态
├── POST /api/trial/register  → 注册转化
│
├── 中间件:
│   ├── Rate Limiting (per IP: 10次/小时)
│   ├── Token Counting (累计 token 使用量)
│   ├── Content Filtering (敏感词过滤)
│   └── Analytics (行为埋点)
│
└── 数据存储:
    ├── Redis (会话缓存、rate limit 计数器)
    └── PostgreSQL (用户注册、使用记录)
```

### 3.4 认证方案

| 用户类型 | 认证方式 | Token 策略 | 安全级别 |
|---------|---------|-----------|---------|
| **匿名试用** | 无需认证，IP 限流 | 一次性 trial-token | 低 (沙盒隔离) |
| **注册用户** | GitHub/Google OAuth | JWT (1小时过期) | 中 |
| **付费用户** | OAuth + 2FA | JWT + refresh token | 高 |

### 3.5 安全措施

| 风险 | 防护措施 |
|------|---------|
| **滥用** | IP rate limit (10次/小时)、CAPTCHA |
| **越权** | Tool Policy 白名单、沙盒隔离 |
| **数据泄露** | 会话自动销毁 (5分钟)、不记录对话内容 |
| **资源耗尽** | Token 上限 (5K/次)、并发限制 |
| **恶意输入** | Content Filtering、prompt 注入防护 |
| **服务可用** | 队列缓冲、熔断机制、降级方案 |

### 3.6 会话管理 (sessions_spawn)

```
试用会话生命周期:

用户点击"试用" 
  → BFF 调用 sessions_spawn 创建隔离会话
  → 配置:
    • runtime: "subagent"
    • model: "glm-5-turbo" (低成本模型)
    • mode: "run" (一次性)
    • sandbox: "require"
    • runTimeoutSeconds: 300 (5分钟)
    • 工具白名单: [read, write, exec(受限)]
  → 返回会话 ID
  → 前端通过 WebSocket 连接会话
  → 用户交互
  → 超时自动销毁
```
