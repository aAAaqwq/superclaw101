# OpenClaw 最佳实践

用好 OpenClaw 不仅仅是配置好工具，更重要的是建立正确的工作流程和习惯。以下是从官方文档和实际使用经验中提炼的核心实践建议。

## Gateway 部署：用 VPS 而不是本地

**Note:**
**最重要的建议：** 不要把 Gateway 运行在你的笔记本电脑上。笔记本会睡眠、会断网、会关机，这些都会导致 OpenClaw 服务中断。


**推荐方案：**

- 使用一台 VPS（虚拟专用服务器）运行 OpenClaw Gateway
- 最低配置：1 核 CPU + 1GB 内存 + 20GB SSD 即可
- 推荐地域：香港或新加坡节点，兼顾国内访问速度和国际 API 延迟
- 使用 systemd 或 PM2 管理 Gateway 进程，确保崩溃自动重启

**为什么 VPS 更好：**

- **24/7 在线**：不会因为你的电脑关机而中断
- **网络稳定**：VPS 通常有更好的国际网络连接
- **Cron 可靠**：定时任务不会因为电脑睡眠而错过执行
- **多设备接入**：手机、电脑、平板都能随时连接同一个 Gateway

## 模型选择：匹配任务复杂度

不同的 AI 任务应该使用不同级别的模型。用一个 Opus 来做简单翻译，就像用大炮打蚊子——浪费钱且没有必要。

| 任务类型 | 推荐模型 | 原因 |
|---------|---------|------|
| 复杂推理、架构设计 | Opus / GLM-5 | 需要深度思考和长链推理 |
| 日常写作、代码生成 | Sonnet / GLM-4.7 | 性价比最优，能力够用 |
| 高频扫描、数据监控 | Haiku / GLM-4-Air | 速度快、成本低，适合大量调用 |

通过合理分配模型，可以将 API 成本降低 50-70%，同时不影响关键任务的输出质量。

## 多 Agent 协作模式

OpenClaw 支持子 Agent（Subagent）机制，你可以并行处理多个任务：

- **主会话**：负责跟用户对话，保持响应速度
- **子 Agent**：处理耗时任务（内容生成、数据分析、代码编写）
- **并行执行**：多个子 Agent 同时工作，互不阻塞

实际案例：同时让 3 个子 Agent 分别处理小红书内容、微信公众号内容、抖音脚本，主会话只负责收集结果并汇报。

## 记忆管理：让 AI 记住重要的事

**Note:**
**核心文件：`MEMORY.md`**。这是 OpenClaw 的长期记忆，session 重启后依然保留。把重要信息写在这里，而不是"记在脑子里"。


**记忆最佳实践：**

1. **及时记录**：做完重要决策后，立即写入 `MEMORY.md`
2. **定期整理**：每周回顾一次，删除过时信息，补充新的洞察
3. **结构化存储**：用标题和分类组织信息，方便快速检索
4. **日记 + 长期记忆**：`memory/YYYY-MM-DD.md` 存原始日志，`MEMORY.md` 存提炼后的精华

## 渠道接入：先测试再上线

接入新的消息渠道（Telegram、Discord、微信公众号等）时，遵循以下流程：

1. **测试环境验证**：先用测试 Bot 或测试频道验证功能
2. **逐步放开**：确认稳定后，再接入生产环境
3. **监控日志**：上线初期密切关注日志，及时发现异常
4. **准备回滚方案**：保留旧配置，出问题可以快速回退

## Cron 任务管理

OpenClaw 的定时任务（Cron）依赖 Gateway 持续运行。如果 Gateway 停了，定时任务也不会触发。

**确保 Cron 可靠的要点：**

- Gateway 必须在 VPS 上 24/7 运行
- 使用进程管理工具（systemd / PM2）自动重启
- 设置健康检查，异常时发送告警
- 时区设置正确（`Asia/Shanghai`），避免定时偏差

```bash
# 检查 Gateway 状态
openclaw gateway status

# 重启 Gateway
openclaw gateway restart
```

## 安全意识

- **API Key 安全**：不要把 API Key 提交到 Git 仓库，使用 `.env` 文件
- **权限最小化**：只给 Agent 需要的权限，不要给 root 级别访问
- **定期轮换**：定期更换 API Key 和访问 Token
- **日志审计**：定期检查 OpenClaw 的操作日志，确保没有异常行为
